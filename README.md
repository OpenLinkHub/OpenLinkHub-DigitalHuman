# OpenLinkHub-DigitalHuman
面向物联网、运维、大屏、客服和行业场景的实时语音交互与数字人智能体。


你的思路非常正确：不要一上来就做“完整数字人平台”，而是先把**实时语音对话闭环**跑通，再逐步叠加**指令执行、数字人驱动、表情手势、多模态感知、业务系统联动**。这样风险最低，也最容易形成可演示、可迭代、可商用的产品路径。

下面给你一套比较完整的分阶段规划，可以作为项目路线图使用。

---

## 一、总体目标定位

OpenLinkHub 生态里可以把这块能力定义为：

> **OpenLinkHub AI Assistant / DigitalHuman：面向物联网、运维、大屏、客服和行业场景的实时语音交互与数字人智能体模块。**

它最终不是单纯的“聊天机器人”，而是一个能做到：

- **听得懂**：语音输入、意图识别、上下文理解；
- **想得明白**：LLM 推理、业务知识问答、任务规划；
- **说得出来**：TTS 实时语音输出；
- **动得起来**：数字人口型、表情、手势、姿态联动；
- **干得了活**：调用设备、系统、API，完成具体操作；
- **接得住业务**：对接 OpenLinkHub IoT、告警、设备、任务、大屏、AI 平台。

整体演进路线可以分为 **6 个阶段**：

```text
阶段 1：实时语音对话闭环
阶段 2：指令识别与工具调用
阶段 3：数字人语音联动
阶段 4：表情、动作、手势联动
阶段 5：业务系统深度集成
阶段 6：多模态智能体与平台化能力
```

---

## 二、阶段 1：实时语音对话闭环

第一阶段的目标很明确：先实现一个“能听、能想、能说、能打断”的实时语音助手。

### 1.1 阶段目标

实现基础实时对话能力：

```text
用户语音输入
    ↓
VAD 检测说话开始/结束
    ↓
ASR 实时转写
    ↓
LLM 理解并生成回复
    ↓
TTS 合成语音
    ↓
前端播放
```

同时支持：

- 实时语音输入；
- ASR 语音识别；
- LLM 回复生成；
- TTS 语音播报；
- 流式输出；
- 用户打断；
- 基础上下文记忆。

---

### 1.2 核心功能清单

#### 语音输入

需要支持：

- 浏览器麦克风采集；
- WebSocket 音频上传；
- 音频格式转换；
- 音频分片传输；
- 噪声过滤基础处理。

建议前端采集格式：

```text
PCM 16kHz / 16bit / mono
```

或者：

```text
Opus / WebRTC Audio
```

---

#### VAD 语音活动检测

VAD 用来判断用户什么时候开始说话、什么时候说完。

核心能力：

- 检测说话开始；
- 检测说话结束；
- 过滤短噪声；
- 避免误触发；
- 为“实时打断”提供判断依据。

可选方案：

| 方案 | 说明 |
|---|---|
| Silero VAD | 开源、效果好、适合本地部署 |
| WebRTC VAD | 轻量、经典、易集成 |
| 服务商 VAD | 云厂商 ASR 通常内置 |

推荐优先：

```text
Silero VAD
```

---

#### ASR 语音识别

第一阶段建议先接成熟方案，别一开始自研。

可选方案：

| 方案 | 类型 | 特点 |
|---|---|---|
| FunASR | 开源 | 中文效果好，支持本地部署 |
| Whisper / Faster-Whisper | 开源 | 多语言强，实时性需优化 |
| 阿里云实时语音识别 | 商业 | 稳定、接入快 |
| 讯飞实时语音识别 | 商业 | 中文强 |
| 火山引擎 ASR | 商业 | 实时能力较成熟 |

如果你重视私有化和开源，推荐：

```text
FunASR + WebSocket 实时识别
```

如果你重视快速落地，推荐：

```text
云 ASR 服务
```

---

#### LLM 回复生成

第一阶段建议支持多模型适配，避免绑定一家。

需要设计统一接口：

```text
/chat/completions
```

支持：

- OpenAI API 兼容接口；
- DeepSeek；
- Qwen；
- GLM；
- Ollama；
- vLLM；
- 本地模型。

LLM 输出建议采用流式：

```text
token by token streaming
```

这样可以边生成边 TTS，降低延迟。

---

#### TTS 语音合成

TTS 是实时体验的关键。建议优先支持流式 TTS。

可选方案：

| 方案 | 类型 | 特点 |
|---|---|---|
| CosyVoice | 开源 | 中文效果好，可私有化 |
| ChatTTS | 开源 | 自然度较好，适合实验 |
| Edge TTS | 免费/非正式 | 接入简单，生产需谨慎 |
| 阿里云 TTS | 商业 | 稳定 |
| 讯飞 TTS | 商业 | 中文成熟 |
| 火山 TTS | 商业 | 音色丰富 |

推荐路线：

- MVP：先接云 TTS 或 Edge TTS；
- 私有化版：接 CosyVoice；
- 高质量商业版：支持第三方 TTS 插件化。

---

### 1.3 实时打断机制

实时打断是体验分水岭。

#### 触发逻辑

当系统正在播放 TTS 时，如果检测到用户开始说话：

```text
用户开始说话
    ↓
VAD 检测到 speech_start
    ↓
立即停止当前 TTS 播放
    ↓
取消当前 LLM 生成
    ↓
清空待播音频队列
    ↓
进入新一轮 ASR
```

#### 后端需要支持

- LLM 请求取消；
- TTS 合成取消；
- 音频队列清空；
- 当前会话状态切换；
- 前端播放中断指令。

会话状态可以设计为：

```text
IDLE
LISTENING
THINKING
SPEAKING
INTERRUPTED
ERROR
```

---

### 1.4 阶段 1 架构建议

```text
Web 前端
  ├── 麦克风采集
  ├── 音频播放
  └── WebSocket 通信
        ↓
Realtime Conversation Server
  ├── VAD
  ├── ASR Adapter
  ├── LLM Adapter
  ├── TTS Adapter
  ├── 会话状态管理
  └── 打断控制
        ↓
ASR / LLM / TTS 服务
```

---

### 1.5 阶段 1 交付物

| 类型 | 交付内容 |
|---|---|
| 前端 Demo | 浏览器语音输入、语音回复播放 |
| 后端服务 | 实时对话 WebSocket 服务 |
| ASR 接入 | 至少一种 ASR 可用 |
| LLM 接入 | 支持 OpenAI 兼容接口 |
| TTS 接入 | 至少一种 TTS 可用 |
| 打断能力 | 用户说话时可中断系统播报 |
| 会话记录 | 保存用户输入和系统回复 |
| 配置文件 | 可配置 ASR/LLM/TTS 提供商 |

## 三、阶段 2：指令识别与工具调用

第二阶段要从“会聊天”升级为“能干活”。

### 2.1 阶段目标

让系统能够识别用户意图，并调用具体工具完成操作。

例如用户说：

```text
查看一下 3 号设备当前状态
```

系统应该能识别为：

```json
{
  "intent": "query_device_status",
  "params": {
    "deviceName": "3号设备"
  }
}
```

然后调用 OpenLinkHub API 查询设备状态，并用自然语言回复。

---

### 2.2 能力设计

#### 意图识别

支持两种方式：

1. **规则/关键词识别**
   - 简单稳定；
   - 适合固定指令；
   - 初期容易落地。

2. **LLM Function Calling / Tool Calling**
   - 灵活；
   - 可处理复杂自然语言；
   - 适合未来扩展。

建议第一版使用：

```text
规则模板 + LLM Tool Calling 混合模式
```

---

### 2.3 工具调用体系

可以设计一个统一工具注册机制。

#### 工具定义示例

```json
{
  "name": "query_device_status",
  "description": "查询指定设备的当前状态",
  "parameters": {
    "type": "object",
    "properties": {
      "deviceId": {
        "type": "string",
        "description": "设备ID"
      }
    },
    "required": ["deviceId"]
  }
}
```

#### 常见工具类型

| 工具类型 | 示例 |
|---|---|
| 设备查询 | 查询设备状态、属性、告警 |
| 设备控制 | 开关设备、调整参数、下发指令 |
| 数据查询 | 查询流量、温度、电量、历史曲线 |
| 告警处理 | 查询告警、确认告警、关闭告警 |
| 任务管理 | 创建任务、查询任务、分配任务 |
| 大屏控制 | 切换大屏模块、展示指定数据 |
| 系统操作 | 查询用户、权限、日志 |

---

### 2.4 操作安全机制

只要涉及“执行操作”，必须加安全控制。

#### 建议分级

| 风险级别 | 操作类型 | 是否需要确认 |
|---|---|---|
| 低风险 | 查询设备状态、查询数据 | 不需要 |
| 中风险 | 确认告警、创建任务 | 需要语音/按钮确认 |
| 高风险 | 控制设备、修改配置、删除数据 | 必须二次确认 + 权限校验 |

例如：

```text
用户：关闭 3 号设备。
助手：该操作会关闭 3 号设备，请确认是否执行。
用户：确认。
助手：已执行关闭操作。
```

---

### 2.5 阶段 2 架构

```text
用户语音
  ↓
ASR
  ↓
LLM
  ↓
Intent / Tool Router
  ├── 设备工具
  ├── 告警工具
  ├── 任务工具
  ├── 大屏工具
  └── 第三方 API 工具
  ↓
业务系统 / OpenLinkHub API
  ↓
LLM 组织回复
  ↓
TTS 播报
```

---

### 2.6 阶段 2 交付物

| 类型 | 交付内容 |
|---|---|
| 意图识别 | 支持常用指令识别 |
| 工具注册 | 可配置工具列表 |
| API 调用 | 可调用 OpenLinkHub 设备/告警 API |
| 权限控制 | 根据用户权限限制工具 |
| 二次确认 | 高风险操作确认机制 |
| 操作日志 | 记录语音指令与执行结果 |
| 错误处理 | 参数缺失、权限不足、执行失败反馈 |

## 四、阶段 3：数字人语音联动

第三阶段开始接入数字人，但先不要一上来做复杂表情手势。先实现：

```text
TTS 语音 → 数字人口型同步 → 视频流输出
```

### 3.1 阶段目标

实现数字人能“开口说话”，与语音回复同步。

链路如下：

```text
LLM 回复文本
    ↓
TTS 生成语音
    ↓
数字人驱动服务
    ↓
口型同步视频生成
    ↓
WebRTC / RTMP / HTTP-FLV 输出
    ↓
前端或大屏播放
```

---

### 3.2 技术方案选择

可选方案：

| 方案 | 适合场景 | 特点 |
|---|---|---|
| Wav2Lip | 原型验证 | 成熟，效果一般 |
| MuseTalk | 实时数字人 | 口型实时性较好 |
| livetalking | 工程框架 | 更适合整合实时链路 |
| SadTalker | 非实时内容 | 一张图驱动，实时性一般 |
| 商业数字人 API | 快速交付 | 效果好但成本高 |

建议：

```text
MVP：livetalking / MuseTalk 跑通实时数字人
后续：按业务需要替换更高质量模型
```

---

### 3.3 视频流输出建议

H5 前端和大屏播放优先考虑：

| 输出方式 | 延迟 | 推荐程度 |
|---|---:|---|
| WebRTC | 最低 | 高 |
| HTTP-FLV | 较低 | 高 |
| HLS/m3u8 | 较高 | 中 |
| RTMP | 浏览器不直接支持 | 中 |

如果是实时对话数字人，推荐：

```text
WebRTC
```

如果是大屏展示、轻交互场景，推荐：

```text
HTTP-FLV
```

---

### 3.4 数字人状态联动

即使暂时没有复杂表情，也要有基础状态：

```text
待机 idle
聆听 listening
思考 thinking
说话 speaking
被打断 interrupted
错误 error
```

前端可以根据状态显示：

- 待机动画；
- 聆听动效；
- 思考动效；
- 说话视频；
- 打断后的停顿动作。

---

### 3.5 阶段 3 交付物

| 类型 | 交付内容 |
|---|---|
| 数字人服务 | 支持音频驱动口型 |
| 视频输出 | 支持 WebRTC 或 HTTP-FLV |
| 前端播放 | 页面可展示数字人视频 |
| 状态同步 | listening/thinking/speaking 状态联动 |
| 打断联动 | 打断时停止 TTS 和数字人说话 |
| 形象配置 | 支持选择默认数字人形象 |
| 性能测试 | 单路数字人延迟、帧率、GPU 占用 |

## 五、阶段 4：表情反馈与手势联动

第四阶段才做“更像人”的表现力。

### 4.1 阶段目标

让数字人不只是嘴动，而是能根据语义、情绪、业务状态产生：

- 表情变化；
- 点头摇头；
- 手势动作；
- 姿态切换；
- 情绪反馈；
- 场景动作。

---

### 4.2 表情联动机制

可以让 LLM 回复时同时输出一个情绪标签。

示例：

```json
{
  "text": "当前设备运行正常，没有发现异常告警。",
  "emotion": "positive",
  "gesture": "nod",
  "expression": "smile"
}
```

常见表情：

| 表情 | 适用场景 |
|---|---|
| neutral | 默认回复 |
| smile | 正常、积极结果 |
| serious | 告警、风险提醒 |
| thinking | 正在分析 |
| surprised | 异常发现 |
| apologetic | 出错、无法完成 |

---

### 4.3 手势动作库

手势不建议完全实时生成，初期可以做动作库。

#### 动作库示例

| 动作 | 含义 |
|---|---|
| nod | 点头确认 |
| shake_head | 摇头否定 |
| wave | 打招呼 |
| point_left | 指向左侧内容 |
| point_right | 指向右侧内容 |
| explain | 讲解手势 |
| alert | 提醒动作 |
| idle_loop | 待机循环 |

---

### 4.4 与业务状态联动

这一步很有价值，特别适合大屏和物联网平台。

例如：

| 业务事件 | 数字人表现 |
|---|---|
| 设备正常 | 微笑、点头 |
| 出现严重告警 | 严肃表情、提醒手势 |
| 正在查询数据 | 思考表情 |
| 操作成功 | 点头确认 |
| 操作失败 | 抱歉表情 |
| 风险预警 | 严肃播报 |

---

### 4.5 阶段 4 交付物

| 类型 | 交付内容 |
|---|---|
| 表情标签 | 支持 LLM 输出 emotion |
| 动作标签 | 支持 gesture/action |
| 动作库 | 至少 5–10 个基础动作 |
| 前端联动 | 根据状态切换动作 |
| 数字人表现 | 口型 + 表情 + 手势同步 |
| 业务映射 | 告警/成功/失败等状态驱动表情 |

---

## 六、阶段 5：业务系统深度集成

第五阶段开始从“数字人演示”变成“行业智能助手”。

### 5.1 阶段目标

与 OpenLinkHub 的核心业务能力深度融合。

让数字人可以：

- 查设备；
- 查告警；
- 查流量；
- 查任务；
- 查日志；
- 控制设备；
- 操作大屏；
- 生成报表；
- 执行运维流程。

---

### 5.2 典型业务场景

#### 设备运维助手

用户：

```text
帮我看一下华北区域离线设备有哪些？
```

助手：

```text
当前华北区域共有 12 台设备离线，其中 5 台离线超过 30 分钟。我已按离线时长排序展示在右侧列表。
```

同时调用：

```text
query_offline_devices(region="华北")
```

---

#### 告警分析助手

用户：

```text
今天有哪些严重告警？
```

助手：

```text
今天共有 8 条严重告警，主要集中在 3 个站点，其中 A 站点告警数量最多。
```

同时大屏跳转：

```text
show_dashboard_panel(panel="alarm_overview")
```

---

#### 大屏讲解员

数字人自动播报：

```text
当前全网设备在线率为 98.6%，较昨日提升 0.4%。流量使用峰值出现在上午 10 点至 11 点。
```

适合：

- 展厅；
- 指挥中心；
- 领导驾驶舱；
- 智慧园区；
- 能源大屏。

---

### 5.3 业务工具分类

| 类别 | 工具示例 |
|---|---|
| 设备工具 | 查询状态、查询属性、下发指令 |
| 告警工具 | 查询告警、确认告警、统计告警 |
| 数据工具 | 查询实时数据、历史趋势、流量统计 |
| 任务工具 | 创建任务、查询任务、绑定设备 |
| 大屏工具 | 切换页面、聚焦模块、展示详情 |
| 报表工具 | 生成日报、导出数据、发送邮件 |
| 知识库工具 | 查询文档、解释故障、推荐处理方案 |

---

### 5.4 阶段 5 交付物

| 类型 | 交付内容 |
|---|---|
| OpenLinkHub API 集成 | 设备、告警、任务、数据接口 |
| 工具权限 | 工具级权限控制 |
| 大屏联动 | 支持语音控制大屏模块 |
| 知识库问答 | 支持接入运维文档 |
| 业务流程 | 支持多步骤任务 |
| 审计日志 | 记录语音操作、工具调用、结果 |
| 场景模板 | 运维助手、大屏讲解员、客服助手 |

---

## 七、阶段 6：多模态智能体与平台化

第六阶段是高级形态，可以做成平台能力。

### 6.1 阶段目标

让系统具备多模态能力和智能体编排能力。

最终形态：

```text
语音 + 文本 + 图像 + 视频 + 设备数据 + 业务系统
                ↓
             智能体
                ↓
      对话 / 分析 / 控制 / 展示 / 执行
```

---

### 6.2 多模态能力

可以逐步支持：

## 十、关键协议设计

### 10.1 前后端 WebSocket 消息

建议统一消息格式。

#### 用户音频上传

```json
{
  "type": "audio.input",
  "sessionId": "s001",
  "data": "base64-audio-chunk",
  "format": "pcm16",
  "sampleRate": 16000
}
```

#### ASR 中间结果

```json
{
  "type": "asr.partial",
  "sessionId": "s001",
  "text": "帮我查一下"
}
```

#### ASR 最终结果

```json
{
  "type": "asr.final",
  "sessionId": "s001",
  "text": "帮我查一下三号设备状态"
}
```

#### LLM 文本流

```json
{
  "type": "llm.delta",
  "sessionId": "s001",
  "text": "三号设备当前"
}
```

#### TTS 音频流

```json
{
  "type": "tts.audio",
  "sessionId": "s001",
  "data": "base64-audio-chunk",
  "format": "mp3"
}
```

#### 数字人动作事件

```json
{
  "type": "avatar.action",
  "sessionId": "s001",
  "action": "nod",
  "expression": "smile",
  "emotion": "positive"
}
```

#### 打断事件

```json
{
  "type": "conversation.interrupt",
  "sessionId": "s001",
  "reason": "user_speech_detected"
}
```

## 十三、第一版最小闭环建议

第一版不要做太多，建议只做这 8 件事：

1. **Web 页面麦克风输入**
2. **VAD 检测用户说话**
3. **ASR 输出识别文本**
4. **LLM 流式生成回复**
5. **TTS 播放语音**
6. **用户说话时打断播放**
7. **识别 3–5 个业务指令**
8. **调用一个真实业务 API**

比如第一版支持这些指令：

```text
查询设备状态
查询今日告警
查询在线设备数量
打开某个大屏页面
生成一段播报内容
```

这样你很快就能形成可演示成果。

## 十五、推荐路线总结

你可以按这个顺序推进：

| 阶段 | 重点 | 目标 |
|---|---|---|
| 1 | 实时语音对话 | 先做到能听、能答、能说、能打断 |
| 2 | 指令识别 | 从聊天升级为能调用工具办事 |
| 3 | 数字人联动 | 让语音回复驱动数字人口型 |
| 4 | 表情手势 | 让数字人有情绪和动作表现 |
| 5 | 业务集成 | 接入设备、告警、大屏、任务等系统 |
| 6 | 平台化 | 做成可配置、多场景、多智能体平台 |

最建议你先做的 MVP 是：

```text
Web 实时语音助手
+ ASR
+ LLM
+ TTS
+ 打断
+ Tool Calling
+ 设备/告警查询 API
```

这个闭环跑通后，再接数字人会非常顺。因为数字人的本质是前面语音智能体的“可视化人格外壳”，核心智能链路必须先稳。
